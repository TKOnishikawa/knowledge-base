## このプレイブックが存在する理由

### 一人で「チーム分」の仕事ができる時代

Claude Codeをはじめとするコーディングエージェントの登場で、一人の人間が従来チーム単位で行っていた作業を実行できるようになった。リサーチ、設計ドキュメント作成、フォルダ構造構築、HTML生成、コード開発 —— これらを対話しながら数時間で完了できる。

しかし、**構造がなければAIとの対話は揮発する**。チャットログは流れ、文脈は断片化し、学びは蓄積しない。

### このプレイブックの目的

1. **再現性**: 次の事業立ち上げで「前回どうやったか」を迷わない
2. **コンテキスト可搬性**: セッションを跨いでも作業が途切れない仕組みを内包
3. **品質の底上げ**: フェーズゲートにより、思いつきではなくエビデンスベースで意思決定

---

## 全体像: 7フェーズ

| Phase | 名前 | 目的 | 所要時間目安 |
|-------|------|------|-------------|
| 0 | コンセプト壁打ち | 事業の核を定義する | 1-2時間 |
| 1 | 徹底リサーチ | エビデンスベースを構築する | 2-4時間 |
| 2 | サービス設計・壁打ち | 理想形→MVP を導出する | 3-6時間 |
| 3 | ドキュメント・システム棚卸し | 必要な全成果物を洗い出す | 1-2時間 |
| 4 | インフラ構築 | 運用基盤をセットアップする | 1-2時間 |
| 5 | ドキュメント詳細構築 | 各文書を専門家品質で作成する | 5-10時間 |
| 6 | ツール・システム開発 | 差別化につながるツールを構築 | 可変 |
| 7 | マーケ・集客・ローンチ | 顧客獲得とサービス開始 | 可変 |

**合計目安**: Phase 0-4 まで2-3セッション（10-16時間）で到達可能。Phase 5以降は事業規模による。

## Phase 0: コンセプト壁打ち

### 目的
事業の核となるコンセプトを定義し、1枚のブリーフにまとめる。

### インプット
- 自分の経験・スキル・想い（暗黙知）
- 市場の課題感（言語化されていなくてもOK）

### アウトプット
- **コンセプトブリーフ**（1ページ）: ターゲット、提供価値、差別化、初期仮説

### 完了条件
- 「誰に」「何を」「なぜ自分が」の3つが1文ずつで言える状態

### Claude Codeでの実行パターン
対話型Q&Aで仮説を引き出し、批判的に検証する。

```
# プロンプトテンプレート例
以下の事業アイデアについて壁打ちしたい。

[アイデアの概要を2-3文で記述]

以下の観点で深掘りしてほしい:
1. ターゲット顧客は具体的に誰か？（ペルソナレベルで）
2. 既存の代替手段は何か？なぜそれでは不十分か？
3. 自分がやる必然性は何か？（他の誰かではダメな理由）
4. 初日から提供できる最小限のサービスは何か？
5. このアイデアが失敗するとしたら、最も確率の高い理由は？

甘い質問はいらない。厳しめに突っ込んでほしい。
```

### 注意点
- この段階では「答え」を出す必要はない。**問いの精度を上げる**ことが目的
- 競合分析はPhase 1で行うので、ここでは深入りしない

## Phase 1: 徹底リサーチ

### 目的
サービス設計の根拠となるエビデンスベースを構築する。

### インプット
- Phase 0のコンセプトブリーフ
- リサーチすべきテーマ一覧（後述のテンプレートで定義）

### アウトプット
- **テーマ別リサーチドキュメント**（Markdown、各2,000-5,000字）
- テーマは通常6-10本

### 完了条件
- 全テーマのリサーチが完了し、構造化されたMarkdownとして保存されている
- 各リサーチに「So What（だから何？）」が明記されている

### Claude Codeでの実行パターン
テーマを事前定義し、Gemini deep_researchで並列実行する。

```
# テーマ定義テンプレート
以下の事業に必要なリサーチテーマを定義する。

事業概要: [コンセプトブリーフから1段落]

テーマを以下のカテゴリで網羅的に洗い出してほしい:
A. 市場・競合（市場規模、主要プレイヤー、トレンド）
B. ターゲット顧客（ニーズ、ペインポイント、購買行動）
C. 専門領域の知見（サービス提供に必要な専門知識）
D. ビジネスモデル（料金体系、収益構造、類似事業の事例）
E. 法規制・コンプライアンス
F. マーケティング・集客チャネル
G-H. [事業固有のテーマ]

各テーマに対して:
- リサーチクエスチョン（3-5個）
- 期待するアウトプットの形式
を定義する。
```

```
# 個別リサーチ実行（/gr Skill 使用時）
テーマB「ターゲット顧客のニーズ分析」についてリサーチを実行する。

リサーチクエスチョン:
1. [具体的な問い]
2. [具体的な問い]
3. [具体的な問い]

英語ソースも含めて調査し、日本市場への示唆を導出してほしい。
出力はMarkdownで、以下の構造にする:
- エグゼクティブサマリー
- 詳細（クエスチョンごと）
- So What（事業設計への示唆）
- 参考文献
```

### 注意点
- リサーチ結果は必ずファイルに保存する（チャットに留めない）
- Content Collectionとして管理すると、後から横断検索・参照が容易
- 各テーマは独立しているので並列実行できる。ただし、テーマ間の関連性は最後に人間が統合する

## Phase 2: サービス設計・壁打ち

### 目的
リサーチに基づいてサービスの理想形を描き、そこからMVPを導出する。

### インプット
- Phase 1の全リサーチドキュメント
- コンセプトブリーフ（Phase 0）

### アウトプット
- **サービス設計書**（HTML、McKinsey Dark Theme推奨）
  - 理想形の全体像
  - MVP定義（初期提供範囲）
  - 料金設計
  - 顧客ジャーニー

### 完了条件
- 「このサービスを明日から売ります」と言えるレベルの具体性がある
- MVP範囲が明確で、Phase 1のエビデンスで裏付けされている

### サブフェーズ構成

| サブフェーズ | 内容 | 所要時間 |
|-------------|------|---------|
| 2a. 理想形設計 | 制約を外してフルスペックを描く | 1-2時間 |
| 2b. 論点整理 | 理想形に対する疑問・課題を洗い出す | 30分-1時間 |
| 2c. MVP導出 | 理想形から初期リリース範囲を切り出す | 1-2時間 |

### Claude Codeでの実行パターン

```
# 2a. 理想形設計
Phase 1のリサーチ結果を踏まえ、[事業名]のサービス設計を行う。

まず「理想形」を描く。制約は考えない。以下を定義してほしい:
1. サービスメニュー全体像（体系図）
2. 各メニューの詳細（目的、内容、想定時間、成果物）
3. 料金設計の選択肢（3パターン）
4. 顧客ジャーニー（認知→検討→契約→実施→継続）
5. 競合との差別化ポイント

Plan modeで構成を固めてから、詳細を書き起こす。
```

```
# 2c. MVP導出
理想形から、以下の基準でMVP範囲を切り出す:
- 初月から提供可能（準備期間1ヶ月以内）
- 一人で運用可能
- 最も顧客価値が高い部分を優先
- 複雑なシステムは後回し（人力で代替可能なものは人力）

MVP に含めるもの / 含めないもの（Phase 2以降に回すもの）を明示する。
```

### 注意点
- **理想形から始める**のが重要。最初からMVPを考えると視野が狭くなる
- HTML出力はGitHub Pagesで公開すると、ステークホルダーへの共有が容易
- 料金設計は「安すぎ」が最大のリスク。エビデンスベースで市場相場を参照する

## Phase 3: ドキュメント・システム棚卸し

### 目的
事業運営に必要な全成果物を洗い出し、優先度付けする。

### インプット
- Phase 2のサービス設計書
- Phase 1のリサーチ（特に法規制・コンプライアンス）

### アウトプット
- **ドキュメント一覧表**（Tier 1 / 2 / 3 に分類）
  - Tier 1: ローンチ前に必須（法的書類、契約書、基本Webページ）
  - Tier 2: ローンチ後1ヶ月以内に必要（運用マニュアル、テンプレート）
  - Tier 3: 事業成長に伴い順次作成（研修資料、高度な分析ツール）

### 完了条件
- 「ローンチに何が足りないか」が一覧で把握できる
- 各ドキュメントに担当（自分 or 外注 or AI）と想定工数が付いている

### Claude Codeでの実行パターン

```
# 棚卸しプロンプト
[事業名]のサービス設計書（添付 or ファイルパス）を分析し、
事業運営に必要な全ドキュメント・システムを網羅的に洗い出す。

カテゴリ:
1. 法務・コンプライアンス（契約書、利用規約、プライバシーポリシー等）
2. 事業計画・財務（事業計画書、収支計画、KPI設計等）
3. サービス運用（マニュアル、テンプレート、チェックリスト等）
4. マーケティング（Webサイト、LP、SNSアカウント、コンテンツ等）
5. 顧客管理（CRM設計、顧客データベース、報告書テンプレート等）
6. 社内システム（ツール選定、データ管理、バックアップ等）

各ドキュメントに対して:
- Tier（1/2/3）
- 想定作成時間
- 作成方法（Claude Code / 専門家依頼 / 自分で手動）
- 依存関係（他のドキュメントが先に必要か）

を付与する。HTML表形式で出力。
```

### 注意点
- **初期ドキュメントだけでなく「運用で蓄積するもの」も含める**（例: セッション記録テンプレート、月次レポート形式）
- 棚卸し結果はHTMLダッシュボードにするとプロジェクト管理に使える
- 完璧を目指さない。Tier 3は「将来やること」の記録であり、今やることではない

## Phase 4: インフラ構築

### 目的
ドキュメントと業務データの管理基盤をセットアップする。

### インプット
- Phase 3のドキュメント一覧
- 事業のフォルダ構造設計

### アウトプット
- **Google Driveフォルダ構造**（テンプレートフォルダ含む）
- **GitHubリポジトリ**（必要に応じて）
- **ツール設定**（会計ソフト、CRM、予約システム等のアカウント作成・初期設定）

### 完了条件
- フォルダ構造が作成され、Tier 1ドキュメントの格納先が全て確保されている
- 命名規則が定義されている

### Claude Codeでの実行パターン

```python
# フォルダ一括作成（PythonスクリプトでDrive APIを使用）
# drive_utils.py を活用してフォルダツリーを自動生成する

folders = {
    "01_商談": ["テンプレート", "顧客別"],
    "02_提案": ["提案書", "見積書"],
    "03_契約": ["契約書", "NDA"],
    "04_デリバリー": ["セッション記録", "成果物", "報告書"],
    "05_マーケティング": ["コンテンツ", "広告素材", "分析"],
    "06_経営管理": ["事業計画", "財務", "KPI"],
    "_templates": [],
}
```

### 設計原則
1. **自己完結性**: 将来の法人化・事業分離に備え、他プロジェクトとフォルダを共有しない
2. **命名規則**: `【クライアント名】YYYY-MM-DD ドキュメント種別` のように統一
3. **テンプレートフォルダ**: `_templates/` に空テンプレートを配置し、新規作成時にコピーする運用
4. **アクセス権設計**: 将来メンバーが増えたときのため、フォルダ単位でアクセス権を設計しておく

### 注意点
- フォルダ構造の変更は後からコストが高い。Phase 3の棚卸しが十分か再確認してから構築する
- Drive APIでの一括作成はスクリプト化しておくと、別事業の立ち上げ時に再利用できる

## Phase 5: ドキュメント詳細構築

### 目的
Phase 3で洗い出した各ドキュメントを、専門家品質で作成する。

### インプット
- Phase 3のドキュメント一覧（Tier順）
- Phase 2のサービス設計書（内容の根拠）
- Phase 1のリサーチ（専門知識の根拠）

### アウトプット
- **Tier 1ドキュメント一式**（最低限、ローンチ前に完成）
- Tier 2は運用開始後に順次作成

### 完了条件
- Tier 1の全ドキュメントが作成され、Phase 4のフォルダに格納されている
- 法務系ドキュメントは専門家レビューの依頼準備ができている

### Claude Codeでの実行パターン

カテゴリごとにバッチ処理する。1ドキュメントずつ Plan mode で仕様を固めてから生成する。

```
# カテゴリ別バッチ処理の例
## 法務カテゴリ（最優先）

以下のドキュメントを順番に作成する。
各ドキュメントについて、まずPlan modeで構成・要件を確認してから本文を生成する。

1. サービス利用規約
   - [事業名]のサービス内容に特化
   - 免責事項、解約条件、知的財産権を含む

2. プライバシーポリシー
   - 個人情報保護法に準拠
   - 取得する情報の種類を具体的に記載

3. 業務委託契約書テンプレート
   - 秘密保持条項を含む
   - 成果物の権利帰属を明記

出力形式: Google Docsにインポート可能なMarkdown
```

### 注意点
- **法務系を最初に作る**。サービス開始後に「利用規約がない」は致命的
- AIで作成した法務ドキュメントは必ず専門家（弁護士・行政書士）のレビューを通す
- 1セッションで全部作ろうとしない。カテゴリ単位で区切り、CLAUDE.mdに進捗を記録する

## Phase 6: ツール・システム開発

### 目的
競争優位につながるカスタムツールを構築する。

### インプット
- Phase 2のサービス設計書（どの部分を自動化・差別化するか）
- Phase 5のドキュメント（運用フローが見えている状態）

### アウトプット
- **カスタムツール / スクリプト**（事業固有のもの）
- **ツール選定ドキュメント**（Buy vs Build の判断記録）

### 完了条件
- MVP提供に必要なツールが動作する状態
- 「作らない」判断も記録されている

### 判断基準: Build vs Buy vs Borrow

| 判断 | 基準 | 例 |
|------|------|-----|
| **Build** | 差別化の源泉 + 市販品では不十分 | 独自アセスメントツール、専用ダッシュボード |
| **Buy** | 標準機能で十分 + 運用コストが低い | 会計ソフト（freee）、予約システム（Calendly） |
| **Borrow** | 無料で利用可能 + カスタマイズ不要 | Google Forms、Notion（無料プラン） |

### Claude Codeでの実行パターン

```
# ツール要件定義
[事業名]のサービス提供において、以下の業務フローを分析する。

[業務フローを記述]

各ステップについて:
1. 手動で行う場合の工数（月間）
2. 自動化による時間削減効果
3. Build / Buy / Borrow の推奨と根拠

を判定する。Build推奨のものについては、技術要件と実装計画を策定する。
```

### 注意点
- **Phase 2-5が完了してからPhase 6に入る**。サービス設計が固まる前にツールを作ると、仕様変更で作り直しになる
- 最初は手動運用でスタートし、ボトルネックが明確になってから自動化する方が効率的
- 「かっこいいツールを作ること」が目的化しないよう注意する

## Phase 7: マーケ・集客・ローンチ

### 目的
最初の顧客を獲得し、サービスを開始する。

### インプット
- Phase 2のサービス設計（ターゲット、チャネル戦略）
- Phase 5の完成ドキュメント（Webサイト、LP等）

### アウトプット
- **ローンチチェックリスト**（Go/No-Go判定）
- **初期マーケティング施策の実行**
- **最初の顧客との契約**

### 完了条件
- 最初の有料顧客が1名以上契約している
- フィードバックループ（顧客→サービス改善）が回り始めている

### Claude Codeでの実行パターン

```
# ローンチチェックリスト生成
[事業名]のローンチ前チェックリストを作成する。

カテゴリ:
1. 法務・コンプライアンス: □ 利用規約 □ プライバシーポリシー □ 契約書テンプレート
2. Webプレゼンス: □ HP □ SNSアカウント □ Googleビジネスプロフィール
3. 集客: □ 初期リード獲得施策 □ 紹介依頼先リスト
4. 運用: □ 予約システム □ 決済手段 □ 初回セッション準備
5. バックオフィス: □ 請求書テンプレート □ 会計ソフト設定

各項目の完了状態を確認し、未完了のものについてアクションプランを策定する。
```

### 注意点
- 完璧を待たない。Tier 1が揃った時点でローンチ可能
- 最初の顧客は「近い人」からでいい。実績とフィードバックを得ることが最優先
- ローンチ後にPhase 5-6を並行して継続する（運用しながら整備するのが現実的）

## コンテキスト可搬性システム

7つのフェーズを実行する上で、最も重要なのは**コンテキストが揮発しない仕組み**を作ることである。

### 3つの層

| 層 | 対象読者 | 形式 | 用途 |
|----|---------|------|------|
| CLAUDE.md | AI（Claude Code） | Markdown | プロジェクトルール、決定ログ、現在の進捗 |
| HTML Dashboard | 人間（自分・ステークホルダー） | HTML（GitHub Pages） | 視覚的な進捗管理、成果物一覧 |
| Knowledge Base | 未来の自分 | Markdown（Astro） | 再利用可能なフレームワーク、学び |

### CLAUDE.md の設計パターン

```markdown
# [事業名] プロジェクト

## 現在のフェーズ
Phase 3: ドキュメント棚卸し（完了）→ Phase 4: インフラ構築（進行中）
```

### セッション間の引き継ぎ

Claude Codeのセッションが切り替わっても、CLAUDE.md / MEMORY.md を読み込むことで以下が即座に復元される:

- 現在どのフェーズにいるか
- 何が完了していて何が未完了か
- 直前のセッションで何をしたか
- 使用しているファイルパスとリソースID

これにより、**「前回の続きから」が1分以内に始められる**。

## 実例: ADHDコーチング事業

このプレイブックは、2026年2月に実施したADHDコーチング事業の立ち上げプロセスを一般化したものである。実績を簡潔に記録する。

### タイムライン

| 日付 | Phase | 実施内容 | 成果物 |
|------|-------|---------|--------|
| 2/23 | 0-2 | コンセプト定義、サービス設計壁打ち | 理想形HTML、MVP設計HTML |
| 2/24 | 2-3 | サービス詳細設計、ドキュメント棚卸し | 棚卸しHTML（84項目）、サービス詳細HTML |
| 2/24 | 4 | Google Driveフォルダ構造構築 | 34フォルダ（omoshiku Drive） |
| 2/25 | 1 | Gemini deep_researchで8テーマ並列リサーチ | 8リサーチドキュメント（adhd-project リポ） |
| 2/25 | 4 | Astro 5リポ構築、GitHub Pagesデプロイ | adhd-project（32ファイル、24ページ） |

### 定量的な成果

- **2セッション**でPhase 0-4を完了（約12時間）
- **4本のHTML設計ドキュメント**をMcKinsey Dark Themeで生成
- **84項目のドキュメント棚卸し**を実施（Tier 1: 28項目、Tier 2: 32項目、Tier 3: 24項目）
- **34のDriveフォルダ**をPython APIスクリプトで一括作成
- **8テーマのリサーチ**をGemini deep_researchで並列実行 → Content Collectionとして構造化
- リサーチ結果をAstro 5サイトとして公開（Pagefind検索付き）

### 得られた教訓

1. **Phase 1（リサーチ）を後回しにしない**: 最初にPhase 0-2を先行したが、リサーチがあった方がサービス設計の精度が格段に上がった。次回はPhase 0→1→2の順序を厳守する
2. **HTMLダッシュボードの威力**: 進捗と成果物を一覧できるHTMLがあると、自分の頭の整理にもなる
3. **フォルダ構造はスクリプト化**: 手動でフォルダを作ると30分以上かかるが、スクリプトなら2分。次の事業でも再利用できる
4. **Content Collectionの活用**: リサーチをMarkdownファイルとして保存し、Astroで構造化すると、検索・参照・公開がシームレス

## アンチパターン（やってはいけないこと）

### 1. リサーチをスキップする

**症状**: 「市場のことはだいたいわかっている」と思い込み、Phase 1を飛ばす。

**結果**: サービス設計が個人の経験則に偏り、バイアスだらけの設計になる。料金設定を間違える。ターゲット顧客のニーズを誤解する。

**対策**: Phase 1は「知っていることの確認」ではなく「知らないことの発見」が目的。最低でも市場規模、競合分析、ターゲットニーズの3テーマはリサーチする。

### 2. ツールを先に作る

**症状**: 技術者にありがちな罠。サービス設計が固まる前に「こんなツールがあれば便利」とPhase 6に着手する。

**結果**: サービス設計が変わるたびにツールの仕様が変わり、作り直しの連続。

**対策**: Phase 6はPhase 2-5の後。最初は手動運用。自動化は「同じ作業を3回以上繰り返した」ことを確認してから。

### 3. 完璧主義に陥る

**症状**: 全てのドキュメントが完璧になるまでローンチしない。Tier 3のドキュメントまで作ろうとする。

**結果**: ローンチが半年遅れる。その間に市場環境が変わる。

**対策**: Tier 1が揃った時点でローンチ可能。Tier 2以降は運用しながら作る。「完璧な準備」より「早いフィードバック」の方が価値が高い。

### 4. コンテキストをチャットに留める

**症状**: Claude Codeとの対話で良い議論ができたが、ファイルに保存せずセッションを閉じる。

**結果**: 次のセッションでは同じ議論をゼロからやり直し。意思決定の根拠が失われる。

**対策**: 重要な意思決定は必ずファイルに永続化する。CLAUDE.mdの決定ログ、設計書HTML、リサーチMarkdown —— 形式は何でもいいが、ファイルとして残す。

### 5. 全フェーズを1セッションでやろうとする

**症状**: 勢いに任せてPhase 0から7まで一気に駆け抜けようとする。

**結果**: 後半のフェーズで集中力が落ち、品質が下がる。特にPhase 5（ドキュメント詳細構築）は質が命。

**対策**: 1セッション = 1-2フェーズが適正。セッション終了時にMEMORY.mdに進捗を記録し、次のセッションの最初に確認する。

## チェックリスト: プレイブックの適用判断

このプレイブックが適しているケース:

- [x] サービス事業（コンサル、コーチング、教育、代行等）の立ち上げ
- [x] 一人または少人数（1-3名）でのスタート
- [x] Claude Codeが使える環境
- [x] GitHub / Google Workspace を利用している
- [x] 「早く始めたいが、構造化もしたい」という意志がある

このプレイブックが適していないケース:

- [ ] 物理製品の製造・販売（サプライチェーン管理は別フレームワークが必要）
- [ ] 大規模組織での新規事業（承認プロセス・予算管理の層が別途必要）
- [ ] 技術プロダクト開発（SaaS等）が主体（プロダクト開発のフレームワークを併用する）
