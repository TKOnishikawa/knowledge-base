Phase 1.1

要件定義 — 上流の判断ミスは下流で10倍の手戻り

データ基盤案件の要件定義は、一般的なアプリ開発とは異なる固有の論点がある。 Data Product Canvas + プレモーテム + AIレビューの三層防御で抜け漏れを防ぐ。

Data Product Canvas — 3つの柱

Desirability

顧客が本当に望んでいるか？

例: 「Connected Sheetsで毎週この数字を見たい」

Feasibility

技術的に実現可能か？

例: 「Salesforceのこのフィールドは取得可能か？」

Viability

ビジネスケースが成立するか？

例: 「このマートの維持コスト vs 意思決定改善効果」

Phase 1.3

実装 — エッジケースを構造的に発見する

実装の品質は「見落としの少なさ」で決まる。チェックリスト駆動 + Defensive SQLパターンで、 非エンジニアでも論点を網羅できる仕組みを構築する。

#チェック項目確認方法重要度
1粒度 (Grain)GROUP BY pk HAVING COUNT(*) > 1必須
2NULLチェックCOUNT(*) - COUNT(col)必須
3値域チェックWHERE col NOT IN (期待値)必須
4参照整合性LEFT JOIN + IS NULL必須
5行数の妥当性前回との行数比較必須
6型の一貫性INFORMATION_SCHEMA推奨

Defensive SQL Patterns
-- NULL安全な集計 COALESCE(SUM(amount), 0) AS total_amount -- ゼロ除算の防止 SAFE_DIVIDE(conversions, clicks) AS conversion_rate -- 重複排除（BQ推奨パターン） SELECT * FROM ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY id ORDER BY updated_at DESC ) AS rn FROM source_table ) QUALIFY rn = 1 -- 型安全なキャスト SAFE_CAST(string_col AS INT64) AS numeric_col

分析SQL 5大アンチパターン

Fan-out JOIN

1対多JOINでSUM/COUNTが膨張する。
対策: JOIN前にサブクエリで集計。

DISTINCTで問題隠蔽

根本原因を隠し、下流で再発。
対策: JOIN条件を修正、重複原因を特定。

SELECT * をビューで使用

スキーマ変更で破損。
対策: カラムを明示的にリスト。

ビューの多段スタック

デバッグ困難。
対策: 重要ロジックはマテリアライズ。

JOINを書いたら「この結合は1対1か？1対多か？」を必ず自問する。 — 黄金ルール
