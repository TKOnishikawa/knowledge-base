Section 2

シークレット管理 — Secret Manager vs 環境変数

本番環境では Secret Manager 一択。環境変数は開発時のみ。IAM制御・監査ログ・自動ローテーションが決め手。

Secret Manager

  • IAM制御で細粒度アクセス管理
  • Cloud Logging / Asset Inventoryで監査可能
  • APIでバージョン管理・自動ローテーション
  • メモリ非露出（プロセスリスト漏洩なし）
VS

環境変数

  • プロセスリスト/ログで漏洩リスク
  • ローテーション時に再デプロイが必要
  • 監査ログなし
  • 即時アクセスのみメリット

実装パターン: SecretCache クラス

# シークレット管理クラス（単一責任原則） class SecretCache: def __init__(self, project_id, refresh_interval=300): # 5分キャッシュ self.client = secretmanager.SecretManagerServiceClient() self.project_id = project_id self._cache = {} def get(self, secret_id, version="latest"): # キャッシュヒット → 即返却 # ミス → API取得 → キャッシュ → 返却 ...
  • GCP ConsoleでSecret Manager有効化
  • gcloud secrets create api-key --data-file=/path/to/key
  • Cloud Run SAに secretmanager.secretAccessor 付与
  • コードで SecretCache を注入
セットアップ所要時間: 約10分 — Secret ManagerはGCPの無料枠に含まれ（10,000アクセス/月）、追加コストはほぼゼロ。
Section 3

レート制限 — 3パターンの使い分け

Exponential Backoff / Circuit Breaker / Queue-basedの3パターンを、エラー種別と負荷状況で選択する。

Exponential Backoff

失敗時に待機時間を指数的に増加（1→2→4秒）+ ジッター

一時的429エラー

Tenacity wait_exponential_jitter

Circuit Breaker

連続失敗でAPI呼び出し停止。回復後にテスト的に再開

カスケード障害防止

Tenacity stop_after_attempt

Queue-based

リクエストをキューで定速処理（Token Bucket）

大量並行リクエスト

throttled-py / asyncio.Queue

Rate Limit Pattern Selection Flow
Q1: 一時的なエラー（429）か?
YES
Exponential Backoff（Tenacity）
NO → Q2へ
↓
Q2: APIが長時間ダウンの可能性?
YES
Circuit Breaker
NO → Q3へ
↓
Q3: 大量の並行リクエストを送るか?
YES
Queue-based（throttled-py）
NO
Exponential Backoff で十分

API別レート制限の具体値

API制限値超過時の挙動
Meta Ads API 動的（使用率ベース） 300秒間ブロック
Meta Insights API 大量取得時タイムアウト 非同期ジョブ利用推奨
Google Ads (mutate) 1リクエスト 10,000オペレーション エラー返却
Google Ads (Audience) 2リクエスト/秒、約200/日/顧客ID エラー返却
Google Ads (コンバージョン) 1リクエスト 2,000件 エラー返却
ベストプラクティス: リトライ回数は3-5回（過剰でSLA違反）。ジッター必須（サーバ負荷分散）。Retry-Afterヘッダを尊重。非リトライエラー（4xx固定）は除外。

ガードレール設計 — エージェントの暴走を4層で防ぐ

マルチエージェントの「暴走」防止には4層のガードレールが必要。$47Kインシデントを二度と起こさないための設計原則。

4-Layer Guardrail Stack
L4: 人間承認（HITL）— 高リスクのみ
L3: 予算制御 — トークン/ステップ/コスト上限
L2: 出力検証 — クリティック/バリデータエージェント
L1: アクション制限 — 許可ツールマトリクス
層対策実装例
L1: アクション制限 許可ツールマトリクス エージェントごとに使用可能ツールを制限
L2: 出力検証 クリティック/バリデータ 出力を別エージェントがレビュー
L3: 予算制御 トークン/ステップ/コスト上限 MAX_STEPS=6、コスト閾値 50/75/90%
L4: 人間承認 HITL（高リスクのみ） 承認チェックポイント挿入
$47K インシデント事例
2つのAIエージェントが再帰ループに陥り、$47,000のAPI使用料が発生。 MAX_STEPS でステップ数上限を設定し、トークン/コスト累計の閾値チェック（50%/75%/90%でアラート）を導入すること。 FINOS AIガバナンスフレームワークの DoW（Denial of Wallet）防止を参照。

ロールバック判断の4基準

1. 影響度

部分失敗がビジネスに与える影響の大きさ。広告費の無駄遣いリスクを評価。

2. 可逆性

作成したリソースをAPI DELETEで削除できるか。メール送信は不可逆。

3. コスト

ロールバック自体のAPI呼び出しコスト。レート制限への影響を考慮。

4. ビジネス要件

アトミック性を求めるか（全成功 or 全失敗）。部分的成功の許容度。
